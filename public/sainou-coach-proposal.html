Part 0 — Why Now
01 — Problem
ADHDの「実行機能」課題は、既存ツールでは解けない

全世界3.66億人の成人ADHDは、「何をすべきかわかっている。でも実行できない」という構造的な課題を抱えている。既存のアプリは個別機能に断片化しており、この課題を根本から解決するものがない。

Critical

5アプリの「ツール疲れ」

朝はRoutinery、日中はStructured、感情はBearable、コーチングはShimmer（$345/月）、集中はFocusmate。5つのアプリの使い分け自体がADHD脳にとって最大のハードルになっている。

Critical

AIコーチングの空白

12アプリを調査した結果、「ユーザーの全体像を継続的に把握し、パーソナライズされたコーチングを提供する」アプリは0。タスク分解AIはあっても、ライフコーチAIは存在しない。

High

「今日のタスク」止まり

既存アプリは「今日何をするか」にフォーカス。人生全体（8領域）のバランスを俯瞰し、長期的な方向性を提案する仕組みがない。タスク消化の繰り返しに意味を見出せなくなる。

High

ドーパミン設計の理論化不足

ゲーミフィケーション（チェック音、ストリーク）はあるが、神経科学に基づいた「ドーパミン予測→タスク配置」のアルゴリズムは市場に全く存在しない。

07 — Technology Architecture
SvelteKit + Capacitor + Supabase。実績ある構成で最速開発。

単一コードベースからPWA（Web）とネイティブアプリ（iOS/Android）を同時デプロイ。本番実績のある構成で、MVPを4週間で完成させる。

Layer 1 — Client

SvelteKit SPA + Capacitor

SvelteKit（adapter-static, ssr=false）でSPA構築。Capacitorでネイティブラップ。PWA版はVercelにデプロイ。

SvelteKit 2.x Svelte 5 (Runes) TypeScript 5.x TailwindCSS 4.x Capacitor 6.x
Layer 2 — Backend

Supabase（Auth + DB + Realtime + Edge Functions）

認証・データベース・リアルタイム通信・サーバーレス関数を一体型で提供。Free枠で個人利用に十分。

PostgreSQL + pgvector Supabase Auth (Google OAuth) Realtime (WebSocket) Edge Functions (Deno) Row Level Security
Layer 3 — AI

Claude API + Gemini Flash（ハイブリッドAI）

コーチング品質が重要な対話はClaude Sonnet/Haiku。ユーティリティ処理（カレンダーパース等）は無料のGemini Flash。

Claude Haiku 4.5 Claude Sonnet 4.6 Gemini 2.0 Flash pgvector (RAG)
Layer 4 — External

Google Calendar API + 音声入力

Google Calendar API v3でスケジュール双方向同期。音声入力はWeb Speech API → Capacitorプラグイン → Gemini Liveの段階導入。

Google Calendar API v3 OAuth 2.0 PKCE Web Speech API Push Notifications

09 — Data Flow
アプリ → Supabase → AI → Google Calendar。全データが繋がる。

ユーザーの入力から、AI処理、外部連携、記憶管理まで。才能解放AIコーチのデータフローの全体像。

Memory — Short

コンテキストウィンドウ

Claude APIの会話履歴。直近の対話内容を保持。セッション終了で揮発。FIFO管理。

Memory — Medium

Supabase DB

今日のチェックイン、エネルギーログ、タスク状態。構造化データとして永続保存。

Memory — Long

pgvector（RAG）

過去の対話サマリー、行動パターン、成功体験をベクトル化。類似度検索で過去の文脈を呼び出し。

Claude Code連携（Phase 3）

Supabase → Google Drive → Claude Codeのブリッジにより、開発者自身がClaude Codeからアプリデータにアクセスし、プロンプト最適化やデータ分析を実行。「開発もアプリの一部」という設計思想。

Appendix A — Research Summary
6調査 + 市場分析の成果
01 — ADHD UXベストプラクティス（44ソース）
主要知見
  • 15秒ルール: 起動→主要機能まで1タップ
  • 選択肢は最大3-4個（決断麻痺回避）
  • マイクロゴール分割で集中持続時間47%向上（RCT）
  • 罪悪感なし通知（Duolingo事例で改善確認）
  • ADHD当事者の80%が1ヶ月以内にアプリを離脱
  • 「アプリを開くこと」自体がハードル → Push通知ワンタップ設計
02 — 競合12アプリ分析
調査対象

Tiimo, Structured, Routinery, Goblin Tools, Fabulous, Finch, Bearable, Focusmate, Shimmer, Inflow, Replika, Pi

決定的発見
  • 「バランスホイール俯瞰」カバー: 0/12
  • 「ドーパミン・スケジューリング」カバー: 0/12
  • 「統合AIコーチング」カバー: 0/12
  • 市場は「第2世代」で飽和 → 「第3世代」の勝者は不在
03 — 技術スタック検証（SvelteKit + Capacitor + Supabase）
検証結果: Go判定
  • 本番実績あり（ジャーナリングアプリ、セルフトラッキングアプリ等）
  • OSSスターターキットも存在
  • 制約: SSR不可（SPA化必須）、iOS SafariのWeb Speech不安定
  • 対策: adapter-static + ssr=false、Capacitorプラグインフォールバック
04 — Google Calendar API統合設計
アーキテクチャ
  • OAuth 2.0 PKCEフローでアクセストークン取得
  • Calendar API v3でイベント読取/書込
  • Google Push Notification → Edge Functions → Realtimeのチェイン
  • FreeBusy APIで空き時間検出
05 — 音声入力パイプライン比較
推奨: 段階的ハイブリッド
  • Phase 1: Web Speech API（$0）→ MVP検証
  • Phase 2: @capawesome-team/capacitor-speech-recognition（$0）
  • Phase 3: Gemini Multimodal Live API（音声AI対話）
  • 結論: Whisper API不要（コスト高 + リアルタイム非対応）
06 — AIコーチング設計パターン
4つの心理学的基盤
  • CBT（認知リフレーミング）→ RSD対応
  • 動機づけ面接法（内発的動機）→ 低ドーパミンタスク
  • ポジティブ心理学（強み活用）→ バランスホイール
  • ACT（受容と行動）→ 先延ばし後の自己批判
記憶管理
  • 短期: コンテキストウィンドウ（FIFO）
  • 中期: Supabase DB（今日のログ）
  • 長期: pgvector（RAG検索、過去パターン）
07 — 市場調査（TAM/SAM/SOM）
市場規模
  • TAM: $18.3B（全世界ADHD成人3.66億人）
  • SAM: $5-8B（先進国スマホ保有者）
  • CAGR: 15.4%（2026-2035）
  • 日本のADHD成人: ~160万人（有病率1.65%）
競合収益ベンチマーク
